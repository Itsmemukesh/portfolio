## 2. Branch Overview

This repo has three branches, each with a distinct purpose:

| Branch | Purpose | Edit manually? |
|--------|---------|----------------|
| `main` | Source code ‚Äî where all your edits live | Yes |
| `feature-new-website` | Feature branch used for the design revamp | Only for new features |
| `gh-pages` | Auto-generated by `npm run deploy` ‚Äî powers the live site | **Never** |

### How the branches relate

```
main  ‚îÄ‚îÄ(npm run deploy)‚îÄ‚îÄ‚ñ∂  gh-pages  ‚îÄ‚îÄ‚ñ∂  GitHub Pages  ‚îÄ‚îÄ‚ñ∂  Live website
 ‚îÇ
 ‚îî‚îÄ‚îÄ feature-new-website  (merge into main when done)
```

### Rules to follow

- **Always make changes on `main`** (or a feature branch, then merge to `main`).
- **Never manually edit `gh-pages`.** It is always overwritten by `npm run deploy`.
- **Always run `npm run deploy` while on `main`.** Running it from any other branch will
  deploy the wrong files to the live site.

## 3. SSH Key Setup ‚Äî Two GitHub Accounts

This laptop has two GitHub accounts configured with separate SSH keys so both can
be used simultaneously without conflicts.

### Accounts

| Account | Usage | SSH Key |
|---------|-------|---------|
| Corporate/work account | Default ‚Äî all work/org repos | `~/.ssh/id_rsa` |
| `Itsmemukesh` (personal) | This portfolio repo | `~/.ssh/id_rsa_personal` |

### How it works ‚Äî `~/.ssh/config`

The SSH config file routes each connection to the correct key based on a **host alias**:

```
# Corporate GitHub account (default)
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_rsa

# Personal GitHub account (Itsmemukesh)
Host github-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_rsa_personal
```

- Work repos use `git@github.com:org/repo.git` ‚Üí picks up `id_rsa` automatically.
- This portfolio uses `git@github-personal:Itsmemukesh/portfolio.git` ‚Üí picks up
  `id_rsa_personal` via the `github-personal` alias.

### Verifying both connections

```bash
ssh -T git@github.com          # Should print: Hi <work-username>!
ssh -T git@github-personal     # Should print: Hi Itsmemukesh!
```

### This repo's remote URL

```bash
git remote -v
# origin  git@github-personal:Itsmemukesh/portfolio.git (fetch)
# origin  git@github-personal:Itsmemukesh/portfolio.git (push)
```

The `github-personal` host alias (not `github.com`) ensures git uses the personal
SSH key for all push/pull operations on this repo.

> **If you ever re-clone this repo on a new machine**, set the remote with:
> ```bash
> git remote set-url origin git@github-personal:Itsmemukesh/portfolio.git
> ```

## 4. Running Locally

### Portfolio only (no build step needed)

```bash
npm start
```

Starts a local server at **http://localhost:3000** using Python's built-in `http.server`.

**Alternative** (if Python is not on PATH):

```bash
npx serve . -p 3000
```

### Blog only (Docusaurus dev server)

```bash
npm run start:blog
```

Starts the Docusaurus dev server at **http://localhost:3000** with hot-reload.
Blog posts live in `blog/blog/`. Any `.md` file you save reloads instantly.

> **Note:** The blog dev server is standalone ‚Äî it doesn't serve the portfolio.
> For a combined preview, run `npm run build` and serve the `_site/` folder.

## 5. Writing a New Blog Post

All blog posts live in `blog/blog/` as Markdown files.

### File naming

```
blog/blog/YYYY-MM-DD-your-post-slug.md
```

Example: `blog/blog/2025-03-01-ai-writing-workflows.md`

### Post template

```markdown
---
slug: ai-writing-workflows
title: How AI Is Changing Technical Writing Workflows
authors:
  name: Mukesh Biswas
  title: Staff Technical Writer
  url: https://Itsmemukesh.github.io/portfolio/
tags: [ai, technical-writing, automation]
date: 2025-03-01
---

Opening paragraph visible in the post listing.

<!-- truncate -->

Rest of the post content goes here. Supports full Markdown and MDX.
```

### Update the homepage preview

After adding a new post, update `blog-posts.json` in the repo root to add it to the
"From the Blog" section on the portfolio homepage:

```json
{
  "title": "How AI Is Changing Technical Writing Workflows",
  "slug": "ai-writing-workflows",
  "date": "2025-03-01",
  "tags": ["AI", "Technical Writing", "Automation"],
  "excerpt": "A one or two sentence summary shown on the portfolio homepage.",
  "readTime": "5 min read",
  "featured": true,
  "category": "AI & Automation",
  "author": "Mukesh Biswas"
}
```

The `slug` in `blog-posts.json` must match the `slug:` in the post frontmatter ‚Äî this
is what links the homepage preview card to the correct Docusaurus page.

## 6. Making and Committing Changes

### Editing content

| What to change | File |
|----------------|------|
| Portfolio page text, sections, structure | `index.html` |
| Colors, fonts, layout, spacing | `style.css` |
| Navigation behavior, animations, blog cards | `app.js` |
| Blog post previews on portfolio homepage | `blog-posts.json` |
| Profile photo | `assets/mukesh_profile_pic.jpg` |
| Blog posts | `blog/blog/YYYY-MM-DD-slug.md` |
| Blog site theme / styles | `blog/src/css/custom.css` |
| Blog site config (title, nav, footer) | `blog/docusaurus.config.js` |

### Commit workflow

```bash
# 1. Check what has changed
git status
git diff

# 2. Stage the changed files
git add index.html style.css app.js
# Or stage everything (excluding node_modules ‚Äî covered by .gitignore)
git add .

# 3. Commit with a clear message
git commit -m "Brief description of what changed"

# 4. Push to GitHub
git push origin main
```

### Using a feature branch (for larger changes)

```bash
# Create and switch to a new feature branch
git checkout -b feature/my-new-section

# ... make your changes, then commit ...
git add .
git commit -m "Add new testimonials section"

# Push the feature branch
git push origin feature/my-new-section

# Merge back into main when ready
git checkout main
git merge feature/my-new-section
git push origin main

# Clean up the feature branch (optional)
git branch -d feature/my-new-section
git push origin --delete feature/my-new-section
```

## 7. Pushing Changes to GitHub

Since this repo uses the `github-personal` SSH alias, push commands work as normal:

```bash
git push origin main
```

Git will automatically use the personal SSH key (`id_rsa_personal`) because the remote
URL contains `github-personal` instead of `github.com`.

### If you get a permission denied error

1. Test the SSH connection first:
   ```bash
   ssh -T git@github-personal
   # Expected: Hi Itsmemukesh! You've successfully authenticated...
   ```

2. If the test fails, your SSH key may not be loaded. Add it:
   ```bash
   ssh-add ~/.ssh/id_rsa_personal
   ```

3. Verify the remote URL is using the alias (not `github.com`):
   ```bash
   git remote -v
   # Should show: git@github-personal:Itsmemukesh/portfolio.git
   ```

   If it shows `github.com` instead, fix it:
   ```bash
   git remote set-url origin git@github-personal:Itsmemukesh/portfolio.git
   ```

## 8. Deploying to the Live Website

> **One command deploys everything** ‚Äî portfolio AND blog.
> Never run `npm run deploy` inside the `blog/` folder directly; that would
> overwrite your portfolio on `gh-pages`.

```bash
# IMPORTANT: always be on main before deploying
git checkout main

# Deploy
npm run deploy
```

### What `npm run deploy` does (step by step)

```
predeploy: npm run build
  ‚îî‚îÄ‚îÄ node scripts/build.js
        1. Installs blog/node_modules (first run only, skipped after)
        2. Runs: npm run build  (inside blog/)  ‚Üí blog/build/
        3. Creates a clean _site/ directory
        4. Copies all portfolio files into _site/
        5. Copies blog/build/ into _site/blog/

deploy: gh-pages -d _site
        Pushes _site/ contents to the gh-pages branch ‚Üí live site
```

Expected output:
```
[build] Building Docusaurus blog...
[build] Assembling _site/...
[build] Copying blog build ‚Üí _site/blog/...
[build] Build complete ‚Üí _site/ is ready for deployment.
Published
```

The live site updates within 1‚Äì2 minutes. Hard refresh if you see stale content: `Ctrl + Shift + R`.

### First-time setup note

The first `npm run deploy` will run `npm install` inside `blog/` automatically
(installs Docusaurus and its dependencies). This takes 2‚Äì3 minutes once.
All subsequent deploys skip this step and are much faster.

### GitHub Pages settings

GitHub Pages must be configured to serve from the `gh-pages` branch:

> GitHub repo ‚Üí **Settings** ‚Üí **Pages** ‚Üí Source: `gh-pages` branch, `/ (root)`

## 10. Troubleshooting

### `npm run deploy` gives "'gh-pages' is not recognized"

Windows is missing the `.cmd` wrapper for the `gh-pages` binary. Fix:

```bash
npm install
npm run deploy
```

### Site did not update after `npm run deploy`

**Most likely cause:** `npm run deploy` was run from the wrong branch.

Check which branch you were on and re-deploy from `main`:

```bash
git checkout main
npm run deploy
```

**Also try:** a hard refresh in the browser (`Ctrl + Shift + R`) since GitHub Pages
has a CDN cache that can serve stale content for a few minutes.

### `403 Permission denied` when pushing

The wrong GitHub account's credentials are being used. Verify:

```bash
ssh -T git@github-personal
# Must say: Hi Itsmemukesh!

git remote -v
# Must show: git@github-personal:Itsmemukesh/portfolio.git
```

If the remote shows `https://` instead of `git@`, switch it to SSH:

```bash
git remote set-url origin git@github-personal:Itsmemukesh/portfolio.git
```

### Work repos accidentally using personal SSH key (or vice versa)

Work repos use `git@github.com:...` and personal repos use `git@github-personal:...`.
As long as the remote URLs use the correct prefix, there is no conflict ‚Äî SSH reads
the `~/.ssh/config` file and routes each connection to the right key automatically.
