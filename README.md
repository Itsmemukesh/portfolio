## 2. Branch Overview

| Branch | Purpose | Edit manually? |
|--------|---------|----------------|
| `main` | Source code — all your edits live here | Yes |
| `feature-new-website` | Feature branch for larger changes | Only for new features |
| `gh-pages` | Auto-generated by `npm run deploy` — never touch | **Never** |

### How the branches relate

```
main  ──(npm run deploy)──▶  gh-pages  ──▶  GitHub Pages  ──▶  Live website
 │
 └── feature-new-website  (merge into main when done)
```

### Rules to follow

- **Always make changes on `main`** (or a feature branch, then merge to `main`).
- **Never manually edit `gh-pages`.** It is always overwritten by `npm run deploy`.
- **Always run `npm run deploy` while on `main`.**
- **After every deploy, run `git checkout main` immediately.** The `gh-pages` tool
  temporarily switches your working branch to `gh-pages` during deployment. If you
  don't switch back, your next commit will accidentally land on `gh-pages`.

```bash
# Safe deploy sequence — always follow this order:
npm run deploy
git checkout main    # ← never skip this step
```

## 3. SSH Key Setup — Two GitHub Accounts

This laptop has two GitHub accounts configured with separate SSH keys.

### Accounts

| Account | Usage | SSH Key |
|---------|-------|---------|
| Corporate/work account | Default — all work/org repos | `~/.ssh/id_rsa` |
| `Itsmemukesh` (personal) | This portfolio repo | `~/.ssh/id_rsa_personal` |

### How it works — `~/.ssh/config`

```
# Corporate GitHub account (default)
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_rsa

# Personal GitHub account (Itsmemukesh)
Host github-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_rsa_personal
```

### Verifying both connections

```bash
ssh -T git@github.com          # Should print: Hi <work-username>!
ssh -T git@github-personal     # Should print: Hi Itsmemukesh!
```

### This repo's remote URL

```bash
git remote -v
# origin  git@github-personal:Itsmemukesh/portfolio.git (fetch)
# origin  git@github-personal:Itsmemukesh/portfolio.git (push)
```

> **If you ever re-clone this repo on a new machine**, set the remote with:
> ```bash
> git remote set-url origin git@github-personal:Itsmemukesh/portfolio.git
> ```

## 4. Running Locally

### Portfolio only (instant, no build needed)

```bash
npm start
# Opens http://localhost:3000 — portfolio homepage
```

Uses Python's built-in HTTP server. The blog preview cards on the homepage will show
but clicking them links to the live site, not localhost. To test the full link chain
locally, use the full site build below.

To exit: press `Ctrl + C`.

### Full site preview (portfolio + blog together)

```bash
npm run build       # Builds Docusaurus + assembles _site/
npm run serve:site  # Serves _site/ at http://localhost:4000
```

Local URLs that match production exactly:

| Local URL | What you see |
|-----------|-------------|
| `http://localhost:4000/portfolio/` | Portfolio homepage |
| `http://localhost:4000/portfolio/blog/` | Blog listing page |
| `http://localhost:4000/portfolio/blog/<slug>` | Individual blog post |

To exit the preview server: press `Ctrl + C`.

### Blog dev server (hot-reload while writing posts)

```bash
npm run start:blog
# Opens http://localhost:3000 — Docusaurus blog only, with live reload
```

> **First run only:** this installs `blog/node_modules/` automatically (~2–3 min).
> All subsequent starts are fast.

This server is blog-only. It does not serve the portfolio. Use it when you are
actively writing or editing blog posts and want instant feedback.

## 5. Writing a New Blog Post

### Step 1 — Create the Markdown file

Add a new file to `blog/blog/` following this naming convention:

```
blog/blog/YYYY-MM-DD-your-post-slug.md
```

Example: `blog/blog/2025-03-01-ai-writing-workflows.md`

### Step 2 — Add the frontmatter

Every post must start with this frontmatter block:

```markdown
---
slug: ai-writing-workflows
title: How AI Is Changing Technical Writing Workflows
authors: [mukesh]
tags: [ai, technical-writing, automation]
date: 2025-03-01
---

First paragraph — visible in the blog listing page as the preview.

<!-- truncate -->

Rest of the post content goes here.
Full Markdown and MDX are supported.
```

**Important:** `authors: [mukesh]` references the `mukesh` key defined in
`blog/blog/authors.yml`. Do not inline author info in posts — use the key.

### Step 3 — Update the portfolio homepage preview

Open `blog-posts.json` in the repo root and add an entry for the new post:

```json
{
  "title": "How AI Is Changing Technical Writing Workflows",
  "slug": "ai-writing-workflows",
  "date": "2025-03-01",
  "tags": ["AI", "Technical Writing", "Automation"],
  "excerpt": "One or two sentence summary shown as the card preview.",
  "readTime": "5 min read",
  "featured": true,
  "category": "AI & Automation",
  "author": "Mukesh Biswas"
}
```

The `slug` value here must exactly match the `slug:` in the post frontmatter.
The homepage renders only the first 3 entries from this file.

### Step 4 — Preview and deploy

```bash
npm run start:blog          # Preview the blog post with hot-reload
# or
npm run build               # Full build to test the link from the portfolio
npm run serve:site          # Then open http://localhost:4000/portfolio/
```

### Adding or updating authors

Author definitions live in `blog/blog/authors.yml`. To add a new author:

```yaml
newauthor:
  name: New Author Name
  title: Their Title
  url: https://their-website.com
  socials:
    linkedin: their-linkedin-handle
    github: their-github-handle
```

Then reference in a post: `authors: [newauthor]`

## 6. Making and Committing Changes

### What to edit

| What you want to change | File(s) |
|-------------------------|---------|
| Portfolio page content, sections, structure | `index.html` |
| Portfolio styles, colors, layout | `style.css` |
| Portfolio nav, animations, blog card rendering | `app.js` |

| Blog site theme / styles | `blog/src/css/custom.css` |
| Blog site config (nav, footer, baseUrl) | `blog/docusaurus.config.js` |
| Build pipeline (how `_site/` is assembled) | `scripts/build.js` |

### Commit workflow

```bash
# 1. Confirm you are on main
git branch   # asterisk (*) should be on main

# 2. Check what has changed
git status
git diff

# 3. Stage changed files
git add .

# 4. Commit
git commit -m "Brief description of what changed"

# 5. Push to GitHub
git push origin main
```

### Using a feature branch (for larger changes)

```bash
git checkout -b feature/my-new-section

# ... make and commit your changes ...

git checkout main
git merge feature/my-new-section
git push origin main

# Clean up (optional)
git branch -d feature/my-new-section
git push origin --delete feature/my-new-section
```

## 8. Deploying to the Live Website

> **One command deploys everything** — portfolio AND blog together.
> Never `cd blog && npm run deploy` — that would overwrite the portfolio on `gh-pages`.

```bash
# Ensure you are on main first
git checkout main

npm run deploy

# Return to main immediately after (see Branch Rules above)
git checkout main
```

### What `npm run deploy` does internally

```
predeploy: node scripts/build.js
  ├── 1. Install blog/node_modules (first run only — skipped after)
  ├── 2. Build Docusaurus  →  blog/build/
  ├── 3. Create _site/portfolio/
  ├── 4. Copy portfolio files  →  _site/portfolio/
  └── 5. Copy blog/build/     →  _site/portfolio/blog/

deploy: gh-pages -d _site/portfolio
  └── Pushes _site/portfolio/ contents to gh-pages branch → live site
```

### Expected output

```
[build] Building Docusaurus blog...
[build] Assembling _site/...
[build] Copying portfolio files → _site/portfolio/...
[build] Copying blog build → _site/portfolio/blog/...
[build] Build complete!
Published
```

### First-time setup note

The very first `npm run deploy` runs `npm install` inside `blog/` automatically.
This installs Docusaurus and its dependencies (~2–3 minutes). All subsequent
deploys skip this step and build much faster.

### GitHub Pages settings

Must be configured once per repo:

> GitHub repo → **Settings** → **Pages** → Source: `gh-pages` branch, `/ (root)`

### Why `_site/portfolio/` and not `_site/`?

The live site URL is `github.io/portfolio/` because GitHub adds the repo name as
a path prefix on project pages. By placing files under `_site/portfolio/` and
deploying with `gh-pages -d _site/portfolio`, all local paths
(`http://localhost:4000/portfolio/...`) match production URLs exactly — no
path mismatch between local testing and the live site.

## 10. Troubleshooting

### Accidentally committed to `gh-pages` instead of `main`

This happens because `npm run deploy` temporarily switches your working branch.
If you forget `git checkout main` after deploying, your next commit lands on `gh-pages`.

**Recover when changes are not yet committed (untracked/unstaged files):**

```bash
git checkout main
# Untracked files carry over to main automatically
```

**Recover when git refuses due to file conflicts with main:**

```bash
git checkout -f main
# Then re-apply your changes manually and commit
```

**Recover when changes are already committed to gh-pages:**

```bash
git log gh-pages --oneline -5   # find your commit hash
git checkout main
git cherry-pick <commit-hash>
git push origin main
# gh-pages will be corrected automatically on the next npm run deploy
```

**Prevention — run this after every deploy without exception:**

```bash
npm run deploy && git checkout main
```

### `npm run deploy` gives "'gh-pages' is not recognized"

Windows is missing the `.cmd` wrapper. Fix:

```bash
npm install
npm run deploy
```

### Blog 404 at `/portfolio/blog/`

The blog build is missing or stale. Force a clean rebuild:

```bash
cd blog
npm run clear    # wipes Docusaurus cache and build
cd ..
npm run build
npm run serve:site
```

### Blog post not appearing on the portfolio homepage

Check two things:

1. The `slug` in `blog-posts.json` exactly matches the `slug:` in the post frontmatter.
2. The post is within the first 3 entries in `blog-posts.json` (only 3 are shown).

### Site did not update after `npm run deploy`

**Most likely cause:** deployed from the wrong branch.

```bash
git checkout main
npm run deploy
git checkout main
```

**Also try:** hard refresh (`Ctrl + Shift + R`) — GitHub Pages CDN can serve
stale content for a few minutes.

### `403 Permission denied` when pushing

```bash
ssh -T git@github-personal          # Must say: Hi Itsmemukesh!
git remote -v                       # Must show: git@github-personal:...

# Fix remote if wrong:
git remote set-url origin git@github-personal:Itsmemukesh/portfolio.git
```

### Work repos accidentally using personal SSH key (or vice versa)

Work repos use `git@github.com:...`, this portfolio uses `git@github-personal:...`.
As long as the remote URLs use the correct prefix, SSH routes each to the right key
automatically via `~/.ssh/config`.
